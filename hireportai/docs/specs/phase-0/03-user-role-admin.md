# SPEC: User Role & Admin RBAC

## Status: Done

## Problem
All authenticated users have identical access. There is no mechanism to
restrict certain endpoints (admin dashboards, card management, analytics,
feature flags) to privileged operators. Without a role system, any
authenticated user can call any route the moment it is wired up ‚Äî every
future admin feature would need ad-hoc protection added on the spot.

## Solution
Add a `role` VARCHAR column (default `"user"`) to the `users` table. Create
a `require_admin` FastAPI dependency that re-uses `get_current_user` and
raises 403 when `role != "admin"`. Expose `role` in `GET /auth/me` so the
frontend knows whether to render admin UI. Promote a user to admin by setting
their `role` directly in the database (no UI in this spec ‚Äî that is Phase 2).

## Acceptance Criteria

- [ ] AC-1: **Schema** ‚Äî Given `alembic upgrade head`, when the migration
  runs against an existing `users` table, then a `role VARCHAR(20) NOT NULL
  DEFAULT 'user'` column is added; every pre-existing row receives the value
  `"user"` without a full-table rewrite; the column is visible in
  `\d users` in psql.

- [ ] AC-2: **ORM default** ‚Äî Given a new user created via
  `get_or_create_user`, when no `role` is supplied, then `user.role == "user"`.

- [ ] AC-3: **`require_admin` blocks non-admins** ‚Äî Given an authenticated
  user with `role = "user"`, when a request is made to any route that
  declares `Depends(require_admin)`, then the response is HTTP 403 with
  `detail: "Admin access required."`.

- [ ] AC-4: **`require_admin` passes admins** ‚Äî Given an authenticated user
  with `role = "admin"`, when a request is made to any route that declares
  `Depends(require_admin)`, then the request proceeds normally (no 403/401).

- [ ] AC-5: **`GET /auth/me` exposes role** ‚Äî Given an authenticated user,
  when `GET /api/v1/auth/me` is called, then the response JSON includes
  `"role": "user"` (or `"admin"`) as a top-level field.

- [ ] AC-6: **Admin ping endpoint** ‚Äî `GET /api/v1/admin/ping` exists,
  requires `Depends(require_admin)`, returns `{"ok": true, "role": "admin"}`
  for admin users and HTTP 403 for non-admins. Serves as a smoke-test target
  for the admin RBAC system.

## API Contract

### `GET /api/v1/auth/me` (updated)
Adds `role` to the response:
```json
{
  "id":         "<UUID string>",
  "email":      "user@example.com",
  "name":       "Display Name",
  "avatar_url": "https://..." | null,
  "role":       "user" | "admin",
  "created_at": "2026-04-07 12:00:00",
  "subscription": { ... }
}
```

### `GET /api/v1/admin/ping` (new)
**Requires:** `Authorization: Bearer <admin access token>`

**Response 200**
```json
{ "ok": true, "role": "admin" }
```

**Response 403** ‚Äî token is valid but user is not admin
```json
{ "detail": "Admin access required." }
```

**Response 401** ‚Äî no token or invalid token

## Data Model Changes

### Migration: `add role column to users`
Generated by `alembic revision --autogenerate -m "add role column to users"`.

```sql
ALTER TABLE users ADD COLUMN role VARCHAR(20) NOT NULL DEFAULT 'user';
```

No index needed ‚Äî role is a low-cardinality filter used only on
`require_admin` checks (one DB round-trip per protected request, same as the
existing user lookup in `get_current_user`).

Downgrade drops the column: `ALTER TABLE users DROP COLUMN role`.

### ORM change (`app/models/user.py`)
```python
from sqlalchemy import text
role: Mapped[str] = mapped_column(
    String(20), server_default=text("'user'"), nullable=False
)
```

`server_default` (not Python-side `default`) ensures the DB applies the
default to existing rows during the `ALTER TABLE` and to any row inserted
without an explicit value.

## UI/UX
None in this spec. The frontend reads `role` from `GET /auth/me` and may
hide/show admin navigation links, but no admin UI pages are built here.
That is Phase 2 (spec `12-admin-card-crud.md`).

## Edge Cases
- **Google login of an admin**: `get_or_create_user` updates mutable fields
  on re-login (name, avatar_url) but does NOT reset `role` ‚Äî a subsequent
  sign-in does not demote an admin back to `"user"`.
- **Unknown role values**: `require_admin` checks `role == "admin"` exactly.
  Any other value (even future `"superadmin"`) is rejected with 403. This
  avoids accidental privilege escalation from typos.
- **JWT does not embed role**: role is checked against the database on every
  request via `get_current_user`, so a role change takes effect immediately
  without requiring a new token.

## Test Plan

### Unit tests (`tests/test_user_roles.py`)
Call the `require_admin` dependency directly (no DB, no HTTP):

- **`test_require_admin_passes_for_admin`**: construct a `User` with
  `role="admin"`, call `await require_admin(user=user)`, assert it returns
  the same user.
- **`test_require_admin_blocks_regular_user`**: construct a `User` with
  `role="user"`, assert `await require_admin(user=user)` raises
  `HTTPException` with `status_code=403`.

### Integration tests (`tests/test_user_roles.py`)
Use the `client` fixture (same pattern as `test_auth.py`):

- **`test_get_me_includes_role`**: sign in via `/auth/google` (mocked),
  call `GET /auth/me`, assert `role == "user"` in response.
- **`test_admin_ping_rejects_regular_user`**: sign in, call
  `GET /api/v1/admin/ping` with bearer token, assert 403.
- **`test_admin_ping_accepts_admin_user`**: sign in, manually update
  `user.role = "admin"` in the test session via SQLAlchemy, call
  `GET /api/v1/admin/ping`, assert 200 with `{"ok": true, "role": "admin"}`.

### Manual verification
1. `alembic upgrade head` ‚Äî check `\d users` shows `role` column.
2. Sign in via the frontend, call `GET /api/v1/auth/me` ‚Äî confirm `role:
   "user"` in response.
3. In psql: `UPDATE users SET role = 'admin' WHERE email = '<your-email>';`
4. Call `GET /api/v1/admin/ping` with a fresh token ‚Äî confirm 200.
5. `python -m pytest tests/ -v --tb=short` ‚Äî all green.
