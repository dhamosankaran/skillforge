# Alembic migration repair notes

## 2026-04-16 — strip duplicate mission CREATEs from c9863b51075d

The migration `c9863b51075d_add_email_preferences_table.py` previously
contained autogenerated `CREATE TABLE` statements for `missions`,
`mission_categories`, and `mission_days`. Those table definitions were
duplicates of what `a4f1d8e73b92_add_missions_and_mission_days_tables.py`
already creates downstream in the chain — and with a worse schema
(no timezone-aware datetimes, no `status` index, no `cards_completed`
default, unlengthened `String` PKs).

The duplicates made `alembic upgrade head` from an empty DB crash with
`DuplicateTableError: relation "missions" already exists` the moment the
chain reached `a4f1d8e73b92`. CI had been failing on this since the
workflow was added.

### What changed

- `c9863b51075d` `upgrade()` now only creates the `email_preferences` table.
- `c9863b51075d` `downgrade()` now only drops `email_preferences`.
- `a4f1d8e73b92` is untouched and remains the canonical mission definition.
- Migration chain order is unchanged (file `down_revision` links preserved).

### Dev DB recovery

CI is unaffected — it always runs from empty.

**If your local dev DB is on revision `c9863b51075d` or later AND was
upgraded before this fix**, the mission tables already exist with the
old (inferior) schema. To reconcile:

```bash
# 1. Drop the accidentally-created mission tables
psql -d hireport -c "DROP TABLE IF EXISTS mission_days, mission_categories, missions CASCADE;"

# 2. Roll alembic's recorded head back to just before c9863b51075d
alembic stamp 802d5ba2e219

# 3. Re-upgrade — this replays c9863b51075d (now email_preferences only)
#    and all downstream migrations including a4f1d8e73b92, which will
#    recreate the mission tables with the canonical schema.
alembic upgrade head
```

If your DB is on a revision *before* `c9863b51075d`, do nothing — a
plain `alembic upgrade head` will land you in the right state.
